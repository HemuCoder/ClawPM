# 项目记忆回写规则

## 目标

让项目记忆在日常推进中持续变清楚，而不是越来越乱。

核心原则：

- `daily` 写短
- `journal` 写细
- `overview` 写稳态

## 什么时候触发回写

当一次项目相关协作形成**稳定阶段结果**时，执行一次回写。

稳定阶段结果包括：

- 完成了一个明确子任务
- 形成了稳定结论
- 发现或解决了关键卡点
- 明确了下一步

以下情况通常不需要回写：

- 只是闲聊
- 只是很短的确认
- 还在发散 brainstorming，结论未稳定
- 与项目无关

## 三层写回规则

### 1. daily：简短摘要

写入 `memory/YYYY-MM-DD.md`。

只记录：

- 今天推进了哪个项目
- 做成了什么
- 形成了什么结论
- 下一步是什么

不要把详细讨论过程写进 daily，避免前一日日记自动加载时浪费上下文。

### 2. journal：详细过程

写入 `memory/topics/<topic>/journal/YYYY-MM.md`。

详细记录：

- 这次具体做了什么
- 为什么这样做
- 讨论了哪些方案
- 定下了什么判断
- 还有什么未决项

journal 是项目过程回放的主要载体。

### 3. overview / 专题文件：长期状态

只在项目全局状态真的变化时，更新：

- `00-overview.md`
- `01/02/03...` 专题文件

适合写入这些文件的变化包括：

- 项目阶段变了
- 目标变了
- 关键卡点变了
- 路线图变了
- 某个专题认知发生稳定变化

## 旧 topic 的处理方式

如果旧 topic 还是单文件或半结构化：

1. 先继续当前任务，不打断用户。
2. 当前任务形成阶段结果后，优先把详细过程写到现有 topic 的可用位置。
3. 再轻量提示后续可整理成结构化 topic。

不要因为结构不理想就擅自停下来迁移。

## 推荐写法

### daily 示例

- InfluenceOS：完成项目记忆 skill 的回写规则设计，确定 daily 简写、journal 详写，下一步整理正式 skill 文案。

### journal 示例

- 讨论了为什么 daily 不应承载详细项目过程：因为次日会自动加载前一日日记，若项目细节过多会在非项目场景白白占用上下文。
- 最终决定将详细推进过程迁移到 topic journal，仅在命中项目时渐进加载。
- 同时明确：项目记忆维护不应完全依赖用户手动提醒，而应在形成稳定阶段结果后自动触发一次回写。
